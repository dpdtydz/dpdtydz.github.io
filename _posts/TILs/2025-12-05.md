### 1. DB Replication Delay로 인한 중복 등록 및 로그인 장애
**발생일**: 2024.11.19 | **장애 시간**: 10:46 - 11:07 (약 21분)

#### 문제 상황
- 신규 유입 회원이 리틀홈런 로그인 시 계정을 찾을 수 없다는 에러 발생
- 특정 벌크 주문 사용자들에게 집중적으로 발생
- 학습기 로그인 실패로 인한 학습 불가능 상태

#### 원인 분석
**1차 원인**: DB Master-Slave Replication Delay
- child/member 테이블은 정상 등록되었으나, subscribe 테이블에 다른 식별자로 등록됨
- edu_littlehomelearn_seq_link 테이블에 동일 사용자에 대한 복수 레코드 생성
- Replication 지연으로 인한 중복 체크 로직 오동작

**2차 원인**: 긴급 수정 중 Bean 주입 오류 (인적 오류)
- DB interface 선언부 교체 누락으로 NPE(NullPointerException) 발생
- 10:23 작업 중 실수, 11:04 Slack 메신저로 확인

#### 해결 과정
1. **10:46** - 장애 최초 인지 (모니터링 알림)
2. **10:50** - 원인 파악 시작 (DB 데이터 분석)
3. **11:04** - 새로운 NPE 발생 지점 확인 (Slack 알림)
4. **11:06** - DB interface 선언부 수정 후 배포
   - lhl-stu-01 auth 서버 배포
   - lhl-stu-02 auth 서버 배포
5. **11:07** - 학습기 로그인 테스트로 이슈 해소 확인

#### 재발 방지 대책
- Master-Slave Replication 모니터링 강화
- 중복 체크 로직에 타임스탬프 기반 검증 추가
- 긴급 배포 시 체크리스트 도입 (Bean 주입 확인 포함)
- 배포 전 통합 테스트 강화

**기술 스택**: MySQL Replication, APM, Slack Monitoring

#### 원인 분석
- Tomcat 단일 Host 설정에서 admin과 auth 서비스가 혼재
- 트래픽 분리 불가능으로 인한 모니터링 어려움
- 한쪽 서비스 장애 시 다른 서비스에도 영향

#### 해결 방안
**정기 PM 작업 (2024.11.01 적용)**
1. **Tomcat 설정 단순화**: Host 2개 → 1개로 통합
2. **포트 분리**:
   - admin 서비스: 7180 포트
   - auth 서비스: 11180 포트
3. **독립적인 upstream 구성**

#### 개선 효과
- 독립적인 로그 관리: admin과 auth 트래픽 분리 추적 가능
- 서비스별 독립 스케일링: 각 서비스별 scale out 가능
- 장애 격리: admin 장애 시 auth 영향 없음
- upstream별 모니터링: 응답시간, 에러율 개별 추적
- 설정 변경 유연성: 한쪽 서비스 설정 변경 시 다른 쪽 영향 없음
- 포트 충돌 방지: 7180과 11180 분리로 동시 운영 가능

**기술 스택**: Tomcat, Nginx, Infrastructure Design

### 3. N+1 쿼리 문제로 인한 성능 저하
**프로젝트**: TinpoGo Server

#### 문제 상황
- 학습 콘텐츠 조회 시 응답 속도 느림
- Step당 4개 카테고리 조회 시 수십 번의 쿼리 발생
- 콘텐츠별 파일 정보 조회로 추가 쿼리 발생

#### 원인 분석
- 기존: Step 개수 × 4 (카테고리) × N (콘텐츠 수) = 수백 번의 쿼리
- Step당 1번 + 콘텐츠당 1번 = 전형적인 N+1 문제
- 파일 정보를 별도 쿼리로 조회

#### 해결 방법
1. **LEFT JOIN 활용**
   - getContentsAllListOfStep: 파일 정보를 한 번에 조회

2. **벌크 조회 구현**
   - 여러 Step의 데이터를 한 번에 조회
   - getCuriosityListOfSteps, getThinkListOfSteps 등

3. **ResultMap 최적화**
   - 파일 정보를 ResultMap에 포함하여 단일 쿼리로 처리

#### 성능 개선 결과
- 개선 후: 카테고리당 1번 = 4번의 쿼리
- 응답 시간 약 80% 단축

**기술 스택**: MyBatis, Query Optimization, LEFT JOIN

### 4. 9주차 학습계획 누락 문제 자동화 해결
**프로젝트**: 리틀홈런 학습 시스템

#### 문제 상황
- 사용자가 로그인 시 9주차 학습계획이 생성되지 않음
- VOC(고객 불만) 발생 및 수동 배치 작업 필요
- 학습 중단으로 인한 사용자 경험 저하

#### 원인 분석
- 기존: 사용자 신고 후 수동 대응 (처리 시간: 수 시간 ~ 1일)
- 8주차까지만 계획이 생성되는 로직 문제

#### 해결 방법
1. **비동기 자동 감지 시스템 구축**
   - 로그인 시 자동으로 9주차 계획 누락 체크
   - 실시간 (10초 내) 자동 생성

2. **방어 코드 및 예외 처리 강화**
   - 중복 실행 방지 로직
   - Slack 모니터링 연동

3. **배치 자동 재시도 메커니즘**
   - 배치 실행 상태를 파일 시스템에 기록
   - 실패 시 자동 재시도
   - 지난 4주간 배치 상태 확인 및 누락 시 알림

#### 비즈니스 임팩트
- **VOC 제로화**: 9주차 학습계획 누락 관련 문의 완전 제거
- **운영 효율**: 수동 배치 작업 불필요
- **사용자 만족도**: 끊김 없는 학습 경험 제공
- **시스템 안정성**: 예외 상황 자동 복구

**기술 스택**: Spring Batch, 비동기 처리, Slack API, 파일 시스템 모니터링

### TinpoGo Server (리틀홈런 틴포고)
**Backend Developer**

학습 콘텐츠 관리 및 회원 학습 데이터 처리 REST API 서버

**프로젝트 개요**
- 패키징: WAR
- 아키텍처: 마이크로서비스 아키텍처 + 레이어드 아키텍처

**주요 기능**
- 학습 콘텐츠 및 단계 관리 API
- Step별 콘텐츠 분류 (호기심, 생각, 이야기1, 이야기2)
- 회원 및 구독 관리
- Spring Batch 기반 배치 작업
- Redis 세션 클러스터 (Session/Data Cluster)
- FCM 푸시 알림
- Circuit Breaker 패턴 (Resilience4j)
- 좋아요/조회수 관리
- 학습 리포트 생성

**기술 스택**
- Backend: Spring Boot 2.7.5, JPA, MyBatis, Java 11 (LTS)
- Database: MySQL 8.0+, Redis, EhCache 2.10.4
- Infrastructure: FCM, AWS, Circuit Breaker
- Tools: Maven, Swagger 2.9.2

**성능 개선 사례**
1. **N+1 쿼리 문제 해결**
   - LEFT JOIN 활용으로 파일 정보 한 번에 조회
   - 벌크 조회로 여러 Step 데이터 동시 처리
   - 기존 수십 번 쿼리 → 4번 쿼리로 개선

2. **Redis 캐싱 전략**
   - 자주 조회되는 데이터 캐싱
   - 응답 시간 50% 단축 및 서버 부하 감소

3. **응답 압축**
   - Gzip 압축 활성화 (최소 1024 bytes)
   - 네트워크 전송량 감소

**주의사항**
- N+1 쿼리 회피: 반드시 DAO 호출 전 데이터 조회
- 성능 저하 요소: 연관 데이터를 한 번에 조회하는 메서드 사용
- 중첩 JSON 응답 지양: 타입 safe한 쿼리 달성을 위해 컴파일러 검증 필수
- SQL Injection 방지: PreparedStatement 사용 (MyBatis 자동 처리)
- XSS 방지: 사용자 입력 데이터 검증
- JWT 토큰 관리: 토큰 만료 시간 명시적 로직 확인
